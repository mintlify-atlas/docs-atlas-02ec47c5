---
title: "GET /messages — Retrieve Messages by Time Range"
sidebarTitle: "GET /messages"
description: "GET /messages — retrieves messages from all subscribed Telegram channels within a specified time range. Filters out service messages and direct messages."
---

This endpoint iterates through all subscribed channel messages using Telethon's `iter_messages`, collecting every message that falls within the specified time window. It filters out `MessageService` objects (e.g. join/leave events, pin notifications) and direct messages (`PeerUser`), returning only channel messages. Each message dict is enriched with a `meta.channel_name` field containing the `@username` of the source channel. Media fields are stripped from every message before the response is returned.

## Endpoint

```
GET /messages
```

## Authentication

Requires an active, authenticated Telethon session on the server. When the server is started with `API_ID=0`, the endpoint automatically falls back to `FakeClient`, which generates a synthetic stream of messages useful for local development and testing.

## Query Parameters

<ParamField query="start_time" type="string" required>
  ISO 8601 datetime string representing the **beginning** of the time window. Messages whose `date` is less than or equal to `start_time` are excluded — the iteration loop breaks as soon as such a message is encountered.

  Example: `2025-01-01T00:00:00Z`
</ParamField>

<ParamField query="end_time" type="string">
  ISO 8601 datetime string representing the **end** of the time window. Passed directly to Telethon's `iter_messages` as `offset_date`, so iteration begins at this point and proceeds backwards in time. Defaults to the current time when omitted.

  Example: `2025-01-02T00:00:00Z`
</ParamField>

## How the Time Range Works

Telethon's `iter_messages` streams messages in **reverse chronological order**, starting from `end_time` and moving backwards toward older messages. The endpoint breaks out of the loop the moment it encounters a message whose `date` is less than or equal to `start_time`.

The practical result is:

- Set `end_time` to the **upper bound** of your window (newest messages you want).
- Set `start_time` to the **lower bound** of your window (oldest messages you want).
- All messages with `start_time < date <= end_time` are returned.

If you omit `end_time`, iteration starts from the current moment, so you will receive all recent messages up until `start_time`.

## Filtering Behavior

<Note>
  The following items are silently skipped and will never appear in the response:

  - **`MessageService` objects** — system events such as member join/leave notifications, message pin events, call records, and similar service entries generated by Telegram itself.
  - **`PeerUser` messages** — any message originating from a direct message (DM) conversation. Only messages belonging to channels are returned.
  - **Media content** — the `media` field is removed from every message dict before the response is serialized. Other message fields that reference media metadata may still be present.
</Note>

## Response

<ResponseField name="data" type="array" required>
  Array of raw Telethon message dicts. Each element corresponds to one channel message that falls within the requested time window. An extra `meta` field is injected by the Telegram Connector before the response is returned.

  <Expandable title="Message object properties">
    <ResponseField name="id" type="integer">
      Unique message ID within its channel, as assigned by Telegram.
    </ResponseField>

    <ResponseField name="date" type="string">
      ISO 8601 timestamp of when the message was sent, including timezone offset. Example: `"2025-01-01T15:30:00+00:00"`.
    </ResponseField>

    <ResponseField name="message" type="string">
      Plain-text content of the message.
    </ResponseField>

    <ResponseField name="peer_id" type="object">
      Identifier of the channel this message belongs to.

      ```json
      { "_": "PeerChannel", "channel_id": 1234567890 }
      ```
    </ResponseField>

    <ResponseField name="from_id" type="object | null">
      Sender identifier. Present when the sender is known; `null` for anonymous channel posts.
    </ResponseField>

    <ResponseField name="reply_to" type="object | null">
      Reply metadata if this message is a reply to another message; `null` otherwise.
    </ResponseField>

    <ResponseField name="entities" type="array | null">
      List of message formatting entities describing bold text, URLs, mentions, hashtags, and similar markup. `null` when no entities are present.
    </ResponseField>

    <ResponseField name="views" type="integer | null">
      Number of times the message has been viewed. `null` when not available.
    </ResponseField>

    <ResponseField name="forwards" type="integer | null">
      Number of times the message has been forwarded. `null` when not available.
    </ResponseField>

    <ResponseField name="reactions" type="object | null">
      Reaction data associated with the message, including counts per emoji. `null` when the message has no reactions.
    </ResponseField>

    <ResponseField name="meta" type="object">
      **Added by Telegram Connector** — not part of the raw Telethon response.

      ```json
      { "channel_name": "examplechannel" }
      ```

      `channel_name` is the `@username` of the channel the message belongs to, resolved from the active dialog list at request time. May be `null` if the channel username cannot be resolved.
    </ResponseField>
  </Expandable>
</ResponseField>

## Status Codes

| Status | Description |
|--------|-------------|
| `200 OK` | Request succeeded. The `data` array contains all matching messages (may be empty if no messages fall within the window). |

## Example Request

<CodeGroup>
```bash cURL
curl 'http://localhost:5004/messages?start_time=2025-01-01T00:00:00Z&end_time=2025-01-02T00:00:00Z'
```
</CodeGroup>

## Example Response

```json
{
  "data": [
    {
      "_": "Message",
      "id": 12345,
      "date": "2025-01-01T15:30:00+00:00",
      "message": "Breaking news: new feature released!",
```

<Warning>
  Large time windows combined with many subscribed channels can produce a very large number of messages. This endpoint does **not paginate** — it collects all matching messages in memory and returns them in a single response. Consider using narrow time windows (e.g., hourly chunks) when querying high-volume channels to avoid excessive memory usage and long response times.
</Warning>

<Note>
  When `API_ID=0`, the server uses `FakeClient` instead of a real Telethon session. `FakeClient` generates an **infinite stream** of synthetic messages and only stops when the loop encounters a message whose `date` falls at or before `start_time`. Always provide a `start_time` that is close to `end_time` (e.g., within a few minutes) in FakeClient mode to prevent the loop from running indefinitely.
</Note>

<Tip>
  To retrieve the latest messages from the past hour, set `start_time` to one hour ago and omit `end_time` entirely — it defaults to the current server time, so you will receive everything published in that window.

  ```bash
  curl "http://localhost:5004/messages?start_time=$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ)"
  ```
</Tip>
